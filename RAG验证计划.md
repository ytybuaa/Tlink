# 认知管理系统 — RAG 核心技术验证计划

## 1. 项目背景

目标产品：一个"认知健身器"——用户记录想法，验证想法，追踪认知演变。
核心能力：语义检索历史想法，按相关性排序，附带验证状态（已推翻/已认同/活跃）。

本次验证不涉及产品开发，只验证一个核心假设：
**RAG 能否在两个异质语料上实现高精度语义检索？**

## 2. 验证目标

### 2.1 主要目标
- 给定一个语义问题，能否从论语/王沪宁日记中检索到**真正相关**的段落？
- 检索结果能否按**相关性从高到低**正确排列？

### 2.2 次要目标（如果时间允许）
- 嵌入向量质量：相似语义的问题是否能命中相同段落？
- 分块策略影响：不同 chunk size 是否影响检索精度？
- 跨语料检索：一个问题是否能同时从两份语料中找到相关答案？

### 2.3 不验证（留给后续）
- 想法状态标记（推翻/认同/修正）
- 矛盾检测
- 模式发现
- 前端交互

## 3. 技术方案

### 3.1 架构

```
用户提问
   ↓
Embedding API（将问题转为向量）
   ↓
Chroma（向量数据库，存储语料块的向量）
   ↓
Top-K 检索结果（按余弦相似度排序）
   ↓
返回最相关的段落
```

### 3.2 Embedding 模型选择（关键风险点）

**首选：Xiaomi MiMo**
- 用户指定使用，价格便宜，OpenAI 兼容
- 风险：MiMo 主要是推理/对话模型，需确认是否提供 `/v1/embeddings` 端点
- 如果 MiMo 有 embedding 接口，直接使用

**备选方案（按优先级排列）：**
1. **DeepSeek**（`deepseek-embedding`）— 有专门的 embedding 模型，OpenAI 兼容
2. **阿里 DashScope**（`text-embedding-v3`）— 中文效果极好，OpenAI 兼容
3. **智谱 GLM**（`embedding-2`）— 中文效果好，OpenAI 兼容
4. **本地模型**（`bge-m3` via sentence-transformers）— 无需 API，但需要 GPU 或较大内存

**验证步骤：** 第一阶段先测试 MiMo 是否有 embedding 端点；如果没有，立即切换到 DeepSeek 或 DashScope。

### 3.3 向量数据库：Chroma

选择理由：
- `pip install chromadb`，零配置
- Python 原生，集成简单
- 支持 metadata 过滤（为后续验证想法状态铺路）
- 本地运行，无外部依赖

### 3.4 分块策略

| 参数 | 值 | 理由 |
|------|------|------|
| chunk_size | 500 字符 | 论语单章+注释通常在 200-500 字；日记段落也在这个范围 |
| chunk_overlap | 100 字符 | 防止跨段落的语义被切断 |
| 分隔方式 | 按自然段落 + 固定长度兜底 | 论语有明确的章节分隔，日记有段落 |

### 3.5 检索策略

- **Top-K = 5**：返回最相关的 5 个段落
- **排序依据**：余弦相似度（cosine similarity）
- **可选增强**：RRF（Reciprocal Rank Fusion）— 如果后续需要多路召回

## 4. 验证语料

### 4.1 论语
- 来源：用户提供的文件
- 特点：古文，语义密度高，每段独立性强
- 验证价值：测试模型对**文言文**语义的理解能力

### 4.2 王沪宁《政治的人生》日记
- 来源：用户提供的文件
- 特点：现代中文，叙事性强，思想隐含在日常记录中
- 验证价值：测试模型对**隐含观点/立场**的语义提取能力

### 4.3 语料差异的意义
这两份语料恰好代表了认知管理系统面对的两种典型场景：
- **论语** = 明确的格言式观点（"己所不欲，勿施于人"）→ 类似用户记录的明确想法
- **王沪宁日记** = 隐含在叙事中的认知（日记中对某个事件的反应暗示了他的认知）→ 类似用户日常碎片中的隐含假设

如果 RAG 能同时处理这两种风格，说明技术方案是 robust 的。

## 5. 评估方法

### 5.1 评估问题集（手动构造）

构造 15-20 个测试问题，每个问题都**有明确的预期答案来源**：

**论语方向（8-10 题）：**
- "孔子如何看待学习与思考的关系？" → 预期命中"学而不思则罔，思而不学则殆"
- "孔子对'仁'的定义是什么？" → 预期命中多处涉及"仁"的章节
- "孔子如何评价管仲？" → 预期命中"管氏而知礼，孰不知礼"及其上下文
- "孔子对朋友之间应该如何相处？" → 预期命中"益者三友，损者三友"

**王沪宁日记方向（8-10 题）：**
- "王沪宁对官僚体系有什么看法？" → 预期命中涉及体制/官僚的段落
- "日记中反映出怎样的读书习惯？" → 预期命中涉及读书/学术的段落
- "王沪宁对国际形势有什么判断？" → 预期命中涉及国际政治的段落

**跨语料方向（2-3 题）：**
- "古代和现代知识分子对治国的思考有什么共同点？" → 预期同时命中论语和日记

### 5.2 评估指标

| 指标 | 定义 | 目标 |
|------|------|------|
| Hit@3 | Top-3 结果中包含预期答案的比例 | ≥ 80% |
| Hit@5 | Top-5 结果中包含预期答案的比例 | ≥ 90% |
| MRR (Mean Reciprocal Rank) | 预期答案在结果中的排名倒数的平均值 | ≥ 0.7 |
| 人工判断 | 结果是否"看起来相关"（不看预期答案） | 主观评估 |

### 5.3 评估流程

1. 跑完所有测试问题，记录每个问题的 Top-5 结果
2. 对每个结果做人工判断：✅ 真相关 / ⚠️ 部分相关 / ❌ 不相关
3. 计算 Hit@3、Hit@5、MRR
4. 输出一份评估报告

## 6. 执行步骤

### Step 1：环境搭建
- 安装依赖：`chromadb`、`openai`（用于调用 API）、`pandas`（用于结果整理）
- 确认 MiMo embedding 端点是否可用（发一个测试请求）

### Step 2：语料预处理
- 读取两个文本文件
- 按分块策略切分为 chunks
- 为每个 chunk 附加 metadata：来源文件名、章节/段落编号、文本内容

### Step 3：构建向量库
- 调用 embedding API，为每个 chunk 生成向量
- 存入 Chroma 数据库
- 确认存储成功，检查数据量

### Step 4：检索测试
- 逐一运行 15-20 个测试问题
- 记录每个问题的 Top-5 结果
- 输出结果到 CSV/JSON 文件

### Step 5：评估与报告
- 人工评估每个结果的相关性
- 计算各项指标
- 写出评估结论和下一步建议

## 7. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| MiMo 没有 embedding 接口 | 无法使用首选模型 | 立即切换到 DeepSeek 或 DashScope |
| 文言文嵌入效果差 | 论语检索精度低 | 尝试加注释/翻译后的文本再嵌入 |
| 分块过大/过小 | 语义被切断或噪声太多 | 调整 chunk_size，跑多组对比 |
| API 限流 | 验证进度慢 | 控制并发，分批处理 |
| 语料文件编码问题 | 读取失败 | 预处理时检测并转换编码 |

## 8. 成功标准

**验证通过（可以继续推进产品开发）：**
- Hit@5 ≥ 80%
- MRR ≥ 0.6
- 人工评估：超过 70% 的结果"看起来相关"

**验证部分通过（需要调整方案）：**
- Hit@5 在 60%-80% 之间
- 论语和日记中有一个明显差于另一个
- 需要调整分块策略或尝试其他 embedding 模型

**验证失败（需要重新评估技术路线）：**
- Hit@5 < 60%
- 说明 embedding 模型或分块策略有根本性问题
- 需要考虑：换模型、换检索方式（如 BM25+向量混合检索）、或重新评估产品设计
